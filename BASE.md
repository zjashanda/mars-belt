# Voice Requirement Test Skill

## Description

该 Skill 用于执行嵌入式设备（CSK芯片）**语音需求功能自动化测试全流程**。\
流程包含：

-   需求 Excel 文档解析，自动生成测试用例和配置文件
-   配置验证与补全（串口/正则/拼音映射/同义词）
-   TTS 音频自动合成（缺失时调用讯飞 API）
-   串口正则表达式自动发现（首次运行时从日志推导）
-   语音功能自动化测试（唤醒 + 命令词识别 + 协议比对）
-   失败用例自动重测
-   CrossTalk 自动分析与 spell2zh 拼音映射修复
-   设备重启自动检测与恢复
-   输出带颜色标注的 xlsx 多 sheet 测试报告

适用于 **嵌入式语音设备需求验证 / 功能回归测试**。

------------------------------------------------------------------------

## Use When

当用户出现以下意图时触发该 Skill：

-   用户说 **"开始语音需求测试"**
-   用户说 **"测试这个需求文档"**
-   用户上传 `.xlsx` 需求文档并要求测试
-   用户要求执行 **语音功能自动化测试**
-   用户要求 **解析需求文档并测试**
-   用户指定 **某个需求 Excel 并开始测试**

# Workflow

## 0. 环境检查（每次测试必须执行）

确认依赖已安装：

    pip install pyserial openpyxl websockets

确认硬件：

-   设备通过 USB 串口连接电脑
-   电脑配备扬声器/音箱（播放音频给设备拾音）
-   音频播放工具：ffplay（首选）/ PowerShell MediaPlayer（Windows自动回退）

------------------------------------------------------------------------

## 1. 解析需求文档，生成测试用例和配置

执行脚本：

    python genTestCase.py -f <需求文档.xlsx> [-b <基础配置.json>]

参数说明：

-   `-f`：需求 Excel 文件路径（必需），Excel 中必须有包含"词条预处理"的 sheet
-   `-b`：基础配置文件（默认 `deviceInfoSUPOER.json`，不存在则生成全新配置）

该脚本负责完成：

-   从"词条预处理" sheet 逐行提取：序号、命令词、泛化词、功能类型、TTS播报、播报模式、发送协议、接收协议
-   按命令词内容自动分类功能模块（开关控制/风速控制/定时功能/摇头控制等）
-   构建 `kw2protocol` 命令词→协议映射
-   同协议词自动归组到 `absorb` 同义词表
-   内置 CHAR_PINYIN 映射表生成拼音，写入 `spell2zh`
-   自动生成声调变体：yi1→yi4（一的变调）、zi→zi3（子的轻声）、bu4→bu2、qi1→qi2、ba1→ba2

输出文件：

-   `testCases.csv` — 测试用例表（11列：用例编号/功能模块/测试类型/命令词/功能类型/期望协议/期望播报/播报模式/优先级/测试方法/预期结果）
-   `deviceInfo_generated.json` — 完整测试配置

------------------------------------------------------------------------

## 2. 检查并补全配置文件

读取 `deviceInfo_generated.json`，逐项验证：

### 必查项

-   **串口配置**：`deviceListInfo.cskApLog.port` 必须为正确的 COM 端口，`baudRate` 通常 115200
-   **正则表达式**：regex 区域保留 key，值可留空（测试时自动发现）

```json
"regex": {
  "rebootReason": ".*Boot Reason: (.*)",
  "endLine": "",
  "wakeKw": "",
  "asrKw": "",
  "sendMsg": "",
  "recvMsg": "",
  "playId": "",
  "volume": ""
}
```

> **关键**: wakeKw 和 asrKw 是不同的正则！wakeKw 捕获 keyword 字段（唤醒词），asrKw 捕获 intentStr 字段（命令词识别结果）。

-   **wordList**：确认包含所有命令词，唤醒词在首位
-   **kw2protocol**：确认有协议的命令词都有正确映射，协议格式 `55 60 XX YY`（YY = (XX + 0xB5) & 0xFF）
-   **spell2zh**：确认拼音映射完整，声调变体已生成
-   **absorb**：确认同协议词已正确归为同义词组（双向：A→[B] 且 B→[A]）
-   **ttsConfig**：确认 `app_id` 和 `api_key` 非空

如果串口配置缺失，需提示用户确认 COM 端口号。

------------------------------------------------------------------------

## 3. 执行语音测试

执行脚本：

    python voiceTestLite.py -f deviceInfo_generated.json -r <数量> -l <标签> [-p <COM端口>]

参数说明：

-   `-f`：配置文件（默认 `deviceInfo_generated.json`）
-   `-r`：测试数量。`0` = 全部命令词，`10` = 前10条（默认值）
-   `-l`：测试标签（结果目录名后缀）
-   `-p`：串口端口覆盖（如 COM14）

该脚本自动完成以下流程：

### 3.1 音频检查

检查 `wavSource/` 下所有命令词的 MP3 是否存在，缺失则调用讯飞 AIUI WebSocket API 自动合成。

### 3.2 串口初始化

连接指定串口，启动后台读取线程，同时写入 `serial_raw.log` 原始日志。

### 3.3 正则自动发现

检查 wakeKw/asrKw/sendMsg/playId 是否缺失，缺失则：

1.  设置 `loglevel 4`
2.  播放唤醒词 → 采集串口日志
3.  唤醒后播放一条有协议的命令词 → 继续采集
4.  用候选模式列表逐一匹配日志行：
    -   wakeKw 候选：`.*keyword\\":\\"(.*?)\\".*intentStr.*` 等，用唤醒词验证
    -   asrKw 候选：`.*intentStr\\":\\"(.*?)\\".*` 等，用命令词验证
    -   sendMsg 候选：`.*send msg:: (.*)` 等，用期望协议验证
    -   playId 候选：`.*play id : (\d+)` 等
5.  验证通过的模式写回配置文件
6.  关闭串口，用新正则重新初始化

### 3.4 首轮测试

遍历命令词列表（**排除唤醒词**，唤醒词在每条测试的唤醒阶段已验证）。

每条命令的测试步骤：

1.  唤醒设备（最多10次）：播放唤醒词 → 等1.5s → 检查 wakeKw 匹配
2.  清空缓冲区 → 播放命令词 → 等3s → 读取 asrKw 识别结果
3.  无结果则重试（最多5次）
4.  收集：ASR原始结果、发送协议、播报ID、接收协议
5.  拼音转汉字（spell2zh）→ 判定结果

判定标准：

| 判定 | 条件 |
|------|------|
| **OK** | ASR 结果 == 命令词，或为其同义词 |
| **CrossTalk** | ASR 有结果，但不匹配且不是同义词 |
| **UnAsr** | 5次尝试后仍无识别结果 |
| **Reboot** | 测试期间检测到设备重启 |

### 3.5 设备重启自动恢复

串口日志中出现 `Boot Reason:` 时：等待5s → 重新 `loglevel 4` → 清除标志 → 继续测试。

### 3.6 失败用例自动重测

首轮完成后，收集所有 CrossTalk + UnAsr 命令词，统一进行第二轮重测。

### 3.7 CrossTalk 自动分析

测试结束后分析所有串扰：

-   ASR 返回拼音 + 协议正确 → **拼音映射缺失（误判）**，自动添加 spell2zh 映射并写回配置
-   ASR 返回汉字 或 协议不一致 → **真实串扰**，记入分析报告

------------------------------------------------------------------------

## 4. 复制修改后配置到结果目录

测试结束后（所有自动修复完成后），将**修改后的配置文件**复制到结果目录。

> 配置文件必须在测试结束后复制，确保包含自动发现的正则和修复的拼音映射。

------------------------------------------------------------------------

## 5. 输出测试结果

测试完成后，结果保存在 `result/<时间戳>_<项目名>_<标签>/` 目录下。

核心输出文件 `testResult.xlsx` 包含 3 个 sheet：

-   **Sheet 1 "首轮测试结果"**：逐条结果（13列），颜色编码：绿=OK，红=CrossTalk，黄=UnAsr，灰=Reboot
-   **Sheet 2 "失败重测结果"**：与 Sheet 1 格式相同，仅含重测结果
-   **Sheet 3 "分析报告"**：测试统计 + CrossTalk 分析（拼音误判/真实串扰明细）+ 自动修复的拼音映射 + 重启记录

------------------------------------------------------------------------

## 6. 报告结果摘要

向用户报告：

-   首轮通过率（OK / 总数）
-   CrossTalk 数（区分拼音误判和真实串扰）
-   UnAsr 数
-   重启次数
-   真实串扰的具体词条和识别结果
-   如有大量拼音误判已自动修复，建议进行第二轮测试

# Key Advantages

该 Skill 具备：

-   **一键式全流程**：需求 Excel → 用例生成 → 测试执行 → 结果分析，一条指令完成
-   **正则自动发现**：首次运行无需手动配置正则，自动从串口日志推导
-   **拼音映射自动修复**：测试后自动补全声调变体，第二次测试更准确
-   **失败自动重测**：CrossTalk + UnAsr 用例自动进行第二轮验证
-   **设备重启自动恢复**：检测重启后自动重设日志等级，无需中断测试
-   **智能串扰分析**：自动区分拼音映射误判和真实串扰，避免误报
-   **同义词智能判定**：同协议词不算串扰，减少误判
-   **xlsx 多 sheet 报告**：首轮结果 + 重测结果 + 分析报告，带颜色标注

# Lessons Learned

实际测试中遇到的关键问题及解决方案：

-   **wakeKw 和 asrKw 必须分开**：两者从同一行日志捕获不同字段（keyword vs intentStr）
-   **配置文件在测试结束后复制**：确保包含自动发现的正则和修复的拼音
-   **唤醒词不当命令词测**：唤醒词在每条命令的唤醒阶段已验证，不需要单独测试
-   **声调变体导致误判**：yi1→yi4、zi→zi3 等变读，genTestCase.py 已内置自动生成
-   **串口占用需 kill 进程**：上次测试未正常退出时用 `taskkill /IM python.exe /F`
