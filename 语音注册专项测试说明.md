## 1. 目的

本文只沉淀一件事：语音注册功能应该怎么测、怎么判、怎么归因。

本文不展开平台打包流程，也不写平台操作命令。重点是让测试人员拿到一包“已经支持语音注册的固件”后，知道：

- 当前配置下哪些语音注册能力应该生效
- 不同配置应该怎么验证
- `VOICE-001 ~ VOICE-014` 分别覆盖什么
- 自动化和人工各自看什么
- 什么属于执行问题，什么属于固件问题

## 2. 什么时候可以测

语音注册不是所有产品都能测。开始前必须先确认：

- 产品能力允许语音注册
  - 例如 `voice_regist=Optional` 或业务明确说明支持
- 当前配置已经打开语音注册
  - `voiceRegEnable=true`
- 固件内实际存在学习配置
  - `firmware.study_config.enable eq true`
- 固件内实际存在可学习目标
  - `firmware.study_config.reg_commands[*]`
  - `firmware.study_config.reg_wakewords[*]`

如果产品本身不支持语音注册：

- 不生成 `VOICE-*` 语音注册结论
- 不把“没有学习能力”记成固件失败
- 结论只能写“当前产品不支持该功能”

## 3. 测试前必须满足的环境

### 3.1 设备链路

常用链路如下：

- 日志/烧录口：`/dev/ttyACM0`
- 控制/BOOT 口：`/dev/ttyACM4`
- 协议口：按当前产品实际接法确定

当前上下电与 boot 控制命令固定为：

- 上下电：`uut-switch1.on` / `uut-switch1.off`
- boot：`uut-switch2.on` / `uut-switch2.off`

如果专项前需要重烧固件，烧录文件也必须固定走一套暂存流程：

1. 清理 `scripts/burn/app.bin`
2. 将待烧录固件复制到 `scripts/burn/app.bin`
3. `Uart_Burn_Tool` 只允许烧录这个暂存后的 `app.bin`

不要把任意外部 `.bin` 路径直接喂给烧录工具。

### 3.2 播报链路

语音注册依赖稳定播报，声卡异常时整套用例都会失真。

当前默认使用的指定声卡为：

- `VID_8765&PID_5678:USB_0_1_3_1_0`

使用规则：

- 优先使用指定声卡
- 指定声卡不可用时，只有系统中仅存在一个 `ListenAI Audio` 设备时才允许回落默认声卡
- 如果存在多个 `ListenAI Audio`，不允许盲退默认声卡，必须报错并要求人工指定

开始语音注册前，先用正式播报链路做最小确认：

1. 用指定声卡播报默认唤醒词 3 次
2. 每次间隔约 3 秒
3. 确认设备有正常唤醒响应和日志

若播报链路通但设备没有正常进入拾音状态，可先在控制串口补发：

- `uut-pa.on`
- `pa-enable.set 0 17 0 1`

### 3.3 用例前置恢复

语音注册很容易被前置用例污染。每条用例开始前必须保证设备回到目标测试环境。

至少要确认：

- 当前学习命令词、学习唤醒词状态符合本用例前置条件
- 当前默认唤醒词或目标唤醒词状态明确
- 没有遗留在“删除中”“学习中”“异常唤醒态”
- 设备没有刚发生异常重启

如果上一条用例改过学习词、唤醒词、删除状态或掉电保持：

- 必须先恢复环境，再执行下一条
- 不能把前置污染造成的失败记到当前用例头上

## 4. 判定总原则

### 4.1 学习类用例必须闭环

语音注册不是“能进学习流程”就算通过。

必须闭环证明：

- 学习成功时，学习词真的可用
- 学习失败时，失败词真的不可用
- 删除成功时，学习词真的失效
- 退出删除或删除失败时，学习词仍然保留
- 默认能力没有被误伤

### 4.2 唤醒判定必须按正式标准

不能只看到一条 `Wakeup` 字样就算唤醒成功。

唤醒通过必须按正式标准判断，至少满足：

- 设备真正进入可识别状态
- 唤醒响应播报正常
- 后续命令窗口可正常使用
- 必要时有协议或意图结果能对上

### 4.3 辅助日志不是决定项

以下日志只作为辅助观察，不是单独的通过/失败条件：

- `refresh config`
- `save config success`

只有在明确做“掉电保持”验证时，它们才用于决定“什么时候可以断电”。最终判定仍然要看：

- 断电后是否真的保持
- 学习词/删除结果/默认能力是否真的符合预期

### 4.4 异常重启优先级最高

语音注册过程里只要不是测试主动上下电引发的重启，一律按 `FAIL` 处理。

典型重启证据：

- 再次出现启动 banner
- 再次出现 `root:/$`
- reboot 计数增加
- boot 签名重新出现

出现重启后：

- 当前用例直接记 `FAIL`
- 不允许被后续恢复正常覆盖掉

## 5. 语音注册配置项怎么验证

本节是本文重点。拿到任意一包支持语音注册的固件后，先看实际配置，再按下表选对应验证动作。

## 5.1 总体映射

| 配置项 | 它控制什么 | 主要验证方式 | 通过标准 | 易误判点 |
| --- | --- | --- | --- | --- |
| `voiceRegEnable` | 语音注册总开关 | 检查是否存在学习入口、学习配置、`VOICE-*` 用例 | 开启时学习功能可进入；关闭时不应出现语音注册能力 | 把“不支持/未开启”误判成固件失败 |
| `releaseRegist.registMode` | 学习模式 | 按该模式对应流程逐阶段验证 | 阶段流转和目标词绑定关系符合模式定义 | 只测“能进流程”，没测目标是否真绑定 |
| `commandRepeatCount` / `wakeupRepeatCount` | 每个学习词需要录几遍 | 做学习成功正例 | 前 `N-1` 轮不终态，第 `N` 轮成功 | 没等播报结束就开始下一轮 |
| `commandRetryCount` / `wakeupRetryCount` | 学习失败允许重试几次 | 做恢复成功和耗尽失败两组 | 重试次数与终态符合配置 | 把一次 `reg again!` 当作最终成功 |
| `commandWordsMinLimit / MaxLimit` | 命令词字数边界 | 做短于下限、等于边界、长于上限三组 | 边界内可学，越界被拒绝 | 用了冲突词导致看起来像字数失败 |
| `wakeupWordsMinLimit / MaxLimit` | 唤醒词字数边界 | 同上 | 边界内可学，越界被拒绝 | 仅看学习提示，不验证学习词是否真生效 |
| `commandSensitivity / wakeupSensitivity` | 学习灵敏度 | 做同环境 A/B 稳定性对比 | 灵敏度变化不破坏流程，趋势符合预期 | 用单次结果去判断灵敏度高低 |
| `commandRegistMaxLimit / wakeupRegistMaxLimit` | 最多能保存多少模板 | 连续学习直到上限，再尝试超限 | 上限内成功，超限时被拒绝或给出限制提示 | 学满后没回测旧词是否仍有效 |
| `reply / replyMode` | 学习流程播报内容和模式 | 逐阶段听播报、对日志、对状态 | 入口/中间/成功/失败/删除阶段播报符合配置 | 只看日志，不看播报是否真的完成 |
| `sndProtocol / recProtocol` | 学习流程协议交互 | 抓串口协议与日志 | 关键阶段协议发收符合配置 | 自己臆造协议，不按配置实际验证 |
| `releaseRegistConfig.*.triggers.*.stages.*` | 阶段触发词与阶段回复矩阵 | 按阶段逐项走流程 | 每个阶段的入口、回复、删除动作都符合配置 | 没读阶段定义，拿固定词瞎测 |

## 5.2 总开关和模式类

### `voiceRegEnable`

验证方法：

1. 先看当前固件是否真的带学习配置
2. 开启时，确认存在学习命令词/学习唤醒词能力
3. 关闭时，确认不会进入语音注册专项执行

通过标准：

- `true` 时，学习链路可执行
- `false` 时，不应生成语音注册主验证结论

### `releaseRegist.registMode`

当前至少要区分两类学习模式：

- `specificLearn`
  - 指定学习
  - 核心是“学习结果要绑定到明确目标”，但是否一定还要补说一遍目标命令，必须看当前阶段配置和实际日志
- `contLearn`
  - 连续学习
  - 常见表现是进入学习后直接出现 `cmdlist get[...]` / `Reg info` / `reg status:1`
  - 当前目标学成后，可能出现 `reg auto next!` 并自动切到下一项

验证方法：

1. 进入学习命令词
2. 先看设备是否已经直接进入 `reg status:1` 学习态
3. 如果没有直接进入学习态，再按当前阶段配置要求补说目标命令
4. 完成学习后，用学习后的别名回测
5. 确认触发的是目标命令，不是其他命令

当前实测必须接受两种真实路径：

- 路径 A：`学习命令词 -> 目标命令 -> reg status:1`
- 路径 B：`学习命令词 -> cmdlist get[...] / Reg info -> reg status:1`

对 `contLearn` 还要额外适配四条收敛规则：

- 如果 `学习命令词` 后已经直接进入 `reg status:1`，后续要直接说注册语料，不再重复补说目标命令
- 如果入口后出现 `reg over!`，优先判定为当前学习模板已满；先做两步 `删除命令词` 清当前模板，再重新进入 `学习命令词`
- 如果 `学习命令词` 已识别成功，但短时间内没有打印 `reg status:1 / cmdlist get / Reg info`，不要立刻判失败；先按直学路径继续播第一遍注册语料，再由后续 `reg again! / reg cmd over success! / reg failed!` 判终态
- `删除命令词` 正例在 `contLearn` 下优先按“删除当前学习词”建模，实测应走两步 `删除命令词 -> 删除命令词`；不要把学习别名直接当删除目标输入，否则设备会把该别名当作已学习命令执行

如果出现 `reg auto next!`，说明当前目标已经学成并自动切到下一项，不应直接判失败。此时应按当前方案决定：

- 继续完成后续项
- 或在确认当前项已学成后，用 `退出学习` 收尾，再回测当前学习词

通过标准：

- 学习别名只绑定当前目标命令
- 不允许出现“学成了，但触发到别的命令”这种串绑
- 不允许把“直入学习态”误判成“没选中目标命令”

不要把 `registMode` 简化成一句“必须先选命令词”。真正决定执行方式的，是：

- `registMode`
- `releaseRegistConfig.*.triggers.*.stages.*`
- 当前日志里真实出现的阶段流转

## 5.3 次数类配置

### `commandRepeatCount` / `wakeupRepeatCount`

这两个参数决定“同一个学习词必须连续录入多少次才算学成”。

验证方法：

1. 进入学习流程
2. 连续录入同一学习词
3. 观察前 `N-1` 轮是否仍处于学习中或继续学习态
4. 观察第 `N` 轮是否进入成功终态
5. 学习成功后再回测该词是否真的生效

通过标准：

- 前 `N-1` 轮不能提前成功
- 第 `N` 轮必须成功
- 成功后学习词真实生效

对应主用例：

- `VOICE-001`
- `VOICE-002`

### `commandRetryCount` / `wakeupRetryCount`

这两个参数决定“学习失败后最多还能重试多少次”。

验证方法必须做两组：

1. 恢复成功组
   - 前面故意录入不一致语料
   - 在允许的重试窗口里重新录对
   - 观察是否最终学习成功
2. 耗尽失败组
   - 故意持续录错直到重试次数耗尽
   - 观察是否最终失败
   - 失败后继续回测该语料必须不生效

通过标准：

- 重试次数与配置一致
- 恢复组最终成功并可用
- 耗尽组最终失败且失败语料不可用

特别注意：

- `wakeupRetryCount=0` 时，第一次不一致后应直接失败，不应还有额外重试
- `commandRetryCount=1` / `wakeupRetryCount=1` 时，第一次错误输入后直接 `reg failed!` 本身就是可接受的正确语义
  - 这类包不能再用“失败后还能继续纠错恢复”的双遍逻辑去判
- `reg length error!` 本身不一定是最终失败
  - 如果后面继续出现 `reg again!` 或仍回到 `reg status:1`，它只是当前轮录入失败
  - 只有继续出现 `error cnt > ...`、`reg failed!`、`reg status:4`，才算终态失败

对应主用例：

- `VOICE-003`
- `VOICE-004`
- `VOICE-005`
- `VOICE-006`

## 5.4 字数边界类配置

### `commandWordsMinLimit / commandWordsMaxLimit`

验证方法：

1. 准备三类非冲突语料
   - 小于下限
   - 等于边界
   - 大于上限
2. 分别进入命令词学习流程测试
3. 对“允许学习”的边界语料继续回测是否真的触发目标命令
4. 对“禁止学习”的越界语料继续回测是否确认未生效

通过标准：

- 小于下限：被拒绝
- 等于下限：允许学习，且学成后可用
- 等于上限：允许学习，且学成后可用
- 大于上限：被拒绝

### `wakeupWordsMinLimit / wakeupWordsMaxLimit`

验证方法与命令词相同，但回测标准必须按正式唤醒标准执行。

特别注意：

- 边界词必须避开默认唤醒词、保留词、功能词
- 否则会把“冲突拦截”误判成“字数校验”

## 5.5 灵敏度类配置

### `commandSensitivity / wakeupSensitivity`

灵敏度不是单次能下结论的参数，必须做同环境对比，不要拿一次成功或一次失败就下结论。

推荐验证方法：

1. 固定同一个人、同一距离、同一环境噪声
2. 选择一组符合字数限制且不冲突的标准测试词
3. 在低灵敏度配置下执行至少 10 到 20 次学习/回测
4. 在高灵敏度配置下执行同样次数
5. 只比较趋势，不比较单次偶发结果

通过标准：

- 灵敏度变化不能破坏基本学习流程
- 高低档之间的拒识/误识趋势应有合理差异
- 不允许因为灵敏度调整导致学习状态机异常、删除异常或重启

如果当前没有可靠的大样本环境，就只做：

- 配置入包正确
- 学习主流程稳定
- 不引入明显系统性异常

不要臆造“灵敏度数值必须达到多少识别率”这类结论。

## 5.6 容量上限类配置

### `commandRegistMaxLimit`

验证方法：

1. 连续学习不同命令词别名，直到达到上限
2. 每学成一个都立即回测一次，确认已保存
3. 达到上限后，再尝试学习第 `N+1` 个
4. 再次回测前面已经学成的词，确认旧模板没有被异常污染

通过标准：

- 上限内的学习词全部可用
- 超限时被拒绝或进入预期限制提示
- 超限后旧学习词仍正常

### `wakeupRegistMaxLimit`

验证方法同上，但回测改为逐个用学习唤醒词做正式唤醒验证。

特别注意：

- 多个学习唤醒词都在时，要按当前配置确认是否允许共存
- 如果有删除再学习场景，删除后必须确认额度真的释放

## 5.7 播报与协议类配置

### `reply`

它决定学习流程里实际播报什么内容。

验证方法：

1. 逐阶段进入学习流程
2. 分别观察入口、继续学习、成功、失败、删除等阶段播报
3. 确认播报与配置意图一致

通过标准：

- 有播报需求的阶段能播报
- 文案含义正确
- 播报结束后设备能继续进入下一阶段

### `replyMode`

它决定学习流程的回复模式。验证时不要只看日志值，还要看设备真实反馈表现。

验证方法：

1. 逐阶段触发
2. 对照当前词条配置确认应是“主”“被”或其他模式
3. 观察日志、协议和真实播报是否一致

通过标准：

- 日志/协议/播报表现与配置一致
- 不出现“日志看起来进了，但用户侧实际上无反馈”的情况

### `sndProtocol / recProtocol`

它们是固定协议配置，不能靠猜。

验证方法：

1. 从实际配置中拿到当前阶段应该发什么、收什么
2. 在学习入口、学习成功、学习失败、删除确认等关键阶段抓协议
3. 对比实际串口输出与配置内容

通过标准：

- 关键阶段协议与配置一致
- 没有漏发、乱发、误发到其他场景

特别注意：

- “查询唤醒词有没有协议”这类结论，必须以当前配置为准
- 没配置协议，就不能凭经验写“应该有返回协议”

## 5.8 阶段配置类

### `releaseRegistConfig.*.triggers.*.stages.*`

这是语音注册里最容易被忽略、也是最不能凭感觉测的部分。

它定义的是：

- 哪句话能进入某个阶段
- 进入后应该播报什么
- 删除场景回复什么
- 某阶段之后应该跳到哪里

验证方法：

1. 先把当前固件的阶段定义读清楚
2. 列出本轮要测的阶段清单
   - 学习入口
   - 目标选择
   - 录词阶段
   - 成功阶段
   - 失败阶段
   - 删除入口
   - 删除确认
   - 退出删除
3. 每个阶段都做“触发词 -> 回复 -> 下一状态”闭环验证
4. 删除场景额外验证 `delReply`

通过标准：

- 触发词能进入目标阶段
- 回复与阶段配置一致
- 状态流转正确
- 删除相关回复与删除结果一致

如果阶段配置改了，测试语料也必须跟着改，不能拿旧词硬测。

## 6. `VOICE-001 ~ VOICE-014` 覆盖关系

`VOICE-*` 不是所有配置都覆盖，但它们覆盖了语音注册最核心的主流程。

其中 `VOICE-003 / VOICE-004` 需要按失败重试次数自适应解释：

- 当 `commandRetryCount / wakeupRetryCount > 1` 时，它们验证的是“失败后进入重试窗口并恢复成功”
- 当 `commandRetryCount / wakeupRetryCount = 1` 时，它们不再验证“恢复成功”，而是验证“第一次失配即退出学习，学习语料不生效”

| 用例 | 核心覆盖点 | 当前主要验证结论 |
| --- | --- | --- |
| `VOICE-001` | 命令词学习次数正例 | `commandRepeatCount` 生效，学成后别名可用 |
| `VOICE-002` | 唤醒词学习次数正例 | `wakeupRepeatCount` 生效，学成后学习唤醒词可用 |
| `VOICE-003` | 命令词失败重试恢复 / 单次失败即退出 | `commandRetryCount>1` 时允许恢复；`commandRetryCount=1` 时首次失配即退出且学习语料不可用 |
| `VOICE-004` | 唤醒词失败重试恢复 / 单次失败即退出 | `wakeupRetryCount>1` 时允许恢复；`wakeupRetryCount=1` 时首次失配即退出且学习唤醒词不可用 |
| `VOICE-005` | 命令词失败重试耗尽 | 重试耗尽后必须失败，失败语料不可用 |
| `VOICE-006` | 唤醒词失败重试耗尽 | 重试耗尽后必须失败，失败唤醒词不可用 |
| `VOICE-007` | 功能命令词冲突 | 功能词不能被学成自定义命令词 |
| `VOICE-008` | 默认唤醒词冲突 | 默认唤醒词不能被学成新唤醒词 |
| `VOICE-009` | 命令词保留词冲突 | 学习/删除/退出等保留词不能被学成命令词 |
| `VOICE-010` | 唤醒词保留词冲突 | 学习/删除/全部删除等保留词不能被学成唤醒词 |
| `VOICE-011` | 删除命令词正例 | 删除后学习命令词失效，默认命令仍正常 |
| `VOICE-012` | 删除命令词反例 | 退出删除后学习命令词仍保留 |
| `VOICE-013` | 删除唤醒词正例 | 删除后学习唤醒词失效，默认唤醒仍正常 |
| `VOICE-014` | 删除唤醒词反例 | 退出删除后学习唤醒词仍保留 |

下面这些配置，`VOICE-001 ~ 014` 不能完全替代，必须额外补专项：

- `commandWordsMinLimit / MaxLimit`
- `wakeupWordsMinLimit / MaxLimit`
- `commandSensitivity / wakeupSensitivity`
- `commandRegistMaxLimit / wakeupRegistMaxLimit`
- `reply / replyMode / sndProtocol / recProtocol`
- `releaseRegistConfig.*.triggers.*.stages.*`

## 7. 自动化当前怎么判

自动化执行器不会只看一条关键词日志。

关键内部判定来自：

- `play stop`
- `reg status: 1`
- `reg again!`
- `reg auto next!`
- `reg cmd over success!`
- `reg over!`
- `reg del`
- `reg simila error!`
- `reg length error!`
- `reg success!`
- `reg failed!`
- `reging user quit process keyword: ...`

当前逻辑要点：

1. 先等学习提示播报结束
2. 再等设备进入学习中状态
3. 每轮录词后判断本轮事件是继续学习、成功还是失败
4. 如果命令词学习出现 `reg auto next!`，说明当前项已学成并自动切到下一项，不能直接判失败
5. 如果只想验证当前项，可以在自动切项后用 `退出学习` 收尾
6. 成功场景必须回测学习词
7. 失败场景必须回测失败词不生效
8. 删除场景必须验证删除后状态
9. 任何意外重启优先记 `FAIL`
10. `contLearn` 命令词学习收敛时，清模板优先走两步 `删除命令词`，不要再用 `删除全部命令词` 破坏其他学习项
11. 进入学习态后，必须等当前提示播报真正结束再说语料；提示未播完就说，当前轮很容易被直接判成失败或长度异常

当前实测还需要理解这些状态值：

- `reg status: 1`
  - 学习中
- `reg status: 3`
  - 学习成功结束
- `reg status: 4`
  - 学习失败结束
- `reg status: 6`
  - 退出学习后的结束态
- `reg status: 7`
  - 删除完成后的结束态

这里最关键的三点：

- 学习提示没播完，不能提前录词
- 唤醒词学习成功后，回测必须按正式唤醒标准，不是只搜 `Wakeup`
- 语音注册阶段内出现的 `algo restart` 不等于整机异常重启，必须和真正的 boot banner 区分
- 删除命令词/删除唤醒词后出现 `algo restart`、`ai create`、`AADC STOP/START` 通常是算法配置刷新，不算失败
- 删除类用例只有在捕获到整机重启证据时才算重启失败，例如 boot banner、`APP version` 重新输出、串口断连重连、非预期上电日志

命令词学习还要额外记住一点：

- `学习命令词 -> 目标命令选择 -> 新学习词录入` 是三个阶段
- 不能把已经进入 `reg status: 1` 后的录词失败，直接记成“目标命令选择失败”
- `contLearn` 下若已看到 `cmdlist get[...]` / `Reg info` / `reg status:1`，说明目标命令选择已经结束，后面阶段只剩新学习词录入
- 历史失败用例收敛时，优先参考 [references/语音注册命令词学习手工验证模板.md](/home/bszheng/.openclaw/skills/mars-belt/references/语音注册命令词学习手工验证模板.md)

## 8. 人工观察重点

人工验证时优先盯下面这些现象：

- 进入学习后，是否先有正确提示播报
- 命令词学习时，是否先完成目标命令选择，再进入新学习词录入阶段
- 播报结束后，设备是否真的进入学习态
- 中间轮是否为继续学习，不是提前终态
- 成功后学习词是否真能用
- 失败后该词是否真不能用
- 删除后学习词是否失效，默认能力是否还在
- 是否出现非预期重启
- 涉及掉电保持时，是否在写入完成后再断电

涉及掉电保持时，辅助做法是：

1. 观察是否出现写配置相关日志
2. 若有，等待其稳定
3. 再额外等待约 10 秒
4. 再断电验证最终是否保持

但最终标准仍然是断电后是否真的生效。

## 9. 当前固定测试语料

当前自动化为了稳定复现，已经固定一组标准语料：

### 9.1 命令词学习

| 用途 | 语料 |
| --- | --- |
| 成功正例 | `笑逐颜开` |
| 失败重试恢复 | `心想事成` |
| 失败耗尽 | `万事大吉` |
| 删除正例 | `一帆风顺` |
| 删除反例保留 | `一生平安` |
| 功能词冲突 | `增大音量` |

### 9.2 唤醒词学习

| 用途 | 语料 |
| --- | --- |
| 成功正例 | `晴空万里` |
| 失败重试恢复 | `小树小树` |
| 失败耗尽 | `小熊维尼` |
| 删除正例 | `你好在吗` |
| 删除反例保留 | `小猫队长` |
| 默认唤醒词冲突 | `小聆小聆` |

### 9.3 保留词冲突

命令词学习保留词：

- `学习唤醒词`
- `删除命令词`
- `退出删除`

唤醒词学习保留词：

- `学习命令词`
- `删除唤醒词`
- `全部删除`

如果当前固件阶段词和这组标准语料不一致，应以当前阶段配置为准，不要强行拿固定词去测阶段流转。

另外要记住一条负例选词规则：

- 语音注册控制词、删除词、退出词不能当普通负例词乱用
- 它们只适用于：
  - 保留词冲突
  - 删除链路
  - 明确要验证“错误输入控制词会耗尽重试次数”的场景
- 其他恢复/耗尽类负例，优先使用平台支持的普通功能词

## 10. 常见误判

### 10.1 只看学习成功播报

错误。必须继续回测学习词是否真的生效。

### 10.2 只看学习失败播报

错误。必须继续回测失败语料是否真的不生效。

### 10.3 只搜到 `Wakeup`

错误。唤醒词必须按正式唤醒标准判定。

### 10.4 不恢复前置环境

错误。前置污染会直接把后续用例带偏。

### 10.5 把辅助日志当决定项

错误。`refresh config`、`save config success` 只是辅助观察项，不是所有用例的决定条件。

### 10.6 自己推协议

错误。协议是否存在、发什么、回什么，都必须以当前配置为准，不能自己脑补。

## 11. 推荐归因方式

### 配置问题

典型特征：

- 产品本身不支持语音注册却硬测
- `voiceRegEnable` 没开
- 固件内没有 `study_config`
- 阶段词、协议、学习目标与测试用例前提不一致

### 执行问题

典型特征：

- 提示播报没结束就开始录词
- 唤醒判定标准错误
- 失败后没继续回测失败词
- 删除后没验证默认能力
- 上一条用例污染了当前环境

### 固件问题

典型特征：

- 学习成功但学习词不可用
- 学习失败但失败词仍可用
- 冲突词未被拦截
- 删除后学习词仍有效
- 删除过程中异常重启
- 阶段回复、状态流转或协议与配置不一致

## 12. 结论

语音注册测试的本质不是“走完一遍学习流程”，而是验证完整状态机：

- 能进入
- 能按配置学习
- 学成后真可用
- 学败后真不可用
- 冲突能拦住
- 删除能闭环
- 默认能力不被破坏
- 全程不异常重启

只要按本文的前置条件、配置项验证方法、`VOICE-*` 覆盖关系和判定原则执行，就能把“不同语音注册配置到底该怎么测”落到可执行层面。
