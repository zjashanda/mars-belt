# Mars Belt 全链路固件验证规则

更新时间：2026-04-13

## 适用范围

以下场景必须先读本文件，再执行：

- 完整产品周测
- 边界值 + 中值打包验证
- 多唤醒、语音注册、掉电保持等依赖型专项
- `FAIL / BLOCK / 重启 / 打包异常` 后的控制变量复测
- 结果目录整理、测试报告生成、邮件发送

若与历史案例文档、旧模板、旧报告口径冲突，以本文件为准。

## 1. 总流程

1. 从当前 skill 根目录 `TOOLS.md` 读取并校验最新 `LISTENAI_TOKEN`
2. 查询当前产品能力、可配置参数、只读观察项、当前版本语言约束
3. 列出本产品：
   - 可打包可验证功能点
   - 可打包但仅能做稳定性观察的功能点
   - 不支持或不在当前产品范围内的功能点
4. 以“最少包数覆盖最多功能”为原则设计打包矩阵
5. 统一复用同一产品标号/周标执行所有包
6. 每个包都执行：
   - 打包
   - 配置断言
   - 烧录
   - 预处理
   - 功能验证
   - 结果归档
7. 若组合包出现 `FAIL`、`BLOCK`、重启或系统性异常，必须追加控制变量包
8. 只有当所有支持功能点都被证明为 `PASS / FAIL / BLOCK` 后，才能发最终邮件

补充要求：

- 长耗时等待场景默认按 `60s` 轮询
- 包括 release 状态查询、自动续跑 watcher、等待新 token 后恢复执行
- 非平台明确限流场景，不再使用数分钟级统一退避

## 2. 测试范围裁剪

### 只测当前产品支持的功能

- 只对当前产品 `Supported / Optional / directly_editable` 的功能点打包验证
- `Unsupported` 功能点不打包、不执行、不写入本产品失败结论
- 只读字段只允许写成“观察项”，不能伪装成主验证项

### 当前已确认的通用裁剪规则

- 任何未执行/裁剪用例必须显式披露
  - 邮件正文和 `weekly/summary.md` 都要写清楚：未测范围、未测原因、由哪个包或哪类验证覆盖
  - 不允许把裁剪后的包描述成“全功能通过”
  - 裁剪项不是失败，但必须透明呈现
- `pkg-02-left-boundary` 在 `timeout=1s` 时只做左边界验证
  - 可以裁剪普通 `CORE-CMD-*` 命令词/全功能用例
  - 原因：自动执行器需等待唤醒确认后再播命令，1s 命令窗口下容易形成时序假失败
  - 报告口径：只验证 timeout 左边界、配置边界、唤醒、播报和串口观察项；命令词全功能由中值稳定包/右边界包覆盖
- `欢迎语 TTS 文案(word)` 不是固件运行验证项
  - 它是页面对服务器合成的验证，不代表固件运行态欢迎语
  - 不要把 `word` 写进固件功能通过结论
- 串口选择默认保持平台默认
  - 常规只验证 `uportBaud` 和 `logLevel`
  - `uportUart / traceUart` 仅在用户明确要求或控制变量定位时单独验证
- `vcn` 必须与产品语言匹配
  - 中文产品只能选中文发音人
  - 英文产品必须用英文发音人
  - 若默认发音人与产品语种匹配，不要为了覆盖而每个包都改发音人

## 3. 打包矩阵设计规则

### 总原则

- 目标是“包数最少、覆盖最多”，不是枚举所有组合
- 大多数产品先按约 `5` 个包设计
- 只有在组合包出现问题后，才允许增加控制变量包
- 不得因为中途失败另起新产品标号

### 参数选值规则

- 数值/枚举项：必须覆盖左边界、中值、右边界
- 字符串项：验证 `1` 次即可
- 布尔项：覆盖 `true/false`
- 依赖项：必须同包验证
- 人工感知项：不单独为“听感/音色”扩包，只要求不导致打包失败、重启、死机或基础功能异常

### 推荐包型

1. 基础中值稳定包
   - 建立可运行基线
   - 覆盖字符串项一次性验证
   - 典型项：`timeout / volLevel / defaultVol / speed / vol / compress` 中值
2. 左边界组合包
   - 覆盖低边界
   - 典型项：`timeout` 最小、`volLevel` 最小、`defaultVol` 最小、`uportBaud` 最小、`logLevel` 最小
3. 右边界组合包
   - 覆盖高边界
   - 典型项：`timeout` 最大、`volLevel` 最大、`defaultVol` 最大、`uportBaud` 最大、`logLevel` 最大、`paConfigEnable=true`
4. 状态/依赖开启包
   - 覆盖依赖功能开启场景
   - 典型项：`multiWkeEnable=true`、`wakeWordSave=true`、`voiceRegEnable=true`（仅支持时）
5. 状态/依赖关闭或隔离包
   - 覆盖另一个布尔值，或做单链路隔离
   - 典型项：`wakeWordSave=false`、`volSave=true/false`、多唤醒单模式包

### 关键依赖

- `wakeWordSave`
  - 必须与 `multiWkeEnable=true` 同测
  - 必须新增 `2` 个额外唤醒词
  - 必须完成切换后再做断电校验
- `voiceRegEnable`
  - 只在当前产品支持时测试
- `multiWkeMode`
  - `specified / loop / protocol` 必须分模式得出结论，不能混成一条
- `volSave`
  - 必须结合默认音量和实际运行音量做断电前后比较
- `speed / vol / compress`
  - 跟随边界值和中值即可，不要求每包切换 `vcn`
- 串口路由
  - 非常规验证项
  - 若确需验证，必须同步本地串口和波特率，否则结论无效

## 4. 功能验证规则

### 通用要求

- 能做配置断言的，先做配置断言
- 能做运行态生效验证的，必须做运行态验证
- 无法直接证明生效的项，至少要证明：
  - 不影响其他配置
  - 不导致设备异常
  - 不导致打包失败、烧录失败、重启或系统性故障

### 基础功能点

- `唤醒时长`
  - 验证配置值已入包
  - 验证超时退出行为符合预期
- `音量档位数`
  - 不能只比配置刻度
  - 必须主动探测真实档位数，再与 `volLevel` 比较
- `初始化默认音量`
  - 必须在被其他音量操作污染前执行
  - 验证初始运行音量是否等于配置预期
- `最大/最小音量上溢下溢播报`
  - 字符串只验证一次即可
  - 重点是边界播报和链路稳定性
- `协议串口波特率`
  - 脚本侧波特率必须同步修改
- `日志级别`
  - 验证低/高日志级别下设备仍能稳定运行
- `音量掉电保存`
  - 调到非默认值后断电重启
  - 对比重启前后音量与默认音量关系
- `合成发音人 / 语速 / 音量 / 压缩比`
  - 重点验证不引入打包失败、重启、播报链路异常
  - 无法自动判断音色时，只做稳定性验证
- `功放配置开关`
  - 验证设备可稳定启动、播报、识别

### 语音注册

- 只在产品支持时执行
- 学习词必须先验证“学得上”
- 学习成功后必须验证“学到的词可用”
- 学习失败后必须验证“该词不可用”
- 删除命令词、删除唤醒词必须做闭环：
  1. 先学习成功
  2. 再验证学习词可用
  3. 执行删除
  4. 删除成功后验证学习词不可用
  5. 若是退出删除或删除失败分支，验证学习词仍可用

### 多唤醒

- 只在产品支持时执行
- 验证前必须新增 `2` 个额外唤醒词
- 必须验证：
  - 默认词
  - 切换到目标词
  - 查询当前词
  - 恢复默认词
  - 非当前词不可继续唤醒
- `wakeWordSave`
  - `true` 时验证断电后保持当前词
  - `false` 时验证断电后恢复默认词

## 5. 重启与系统性异常处理

### 重启优先级最高

- 先区分：
  - 用例主动断电重启
  - 设备测试过程中自行重启
- 任何设备自行重启一律 `FAIL`
- 不得因后续恢复正常而把重启包记成通过

### 发现重启后的动作

1. 记录触发动作、触发 case、日志位置
2. 停止把该问题当普通识别失败处理
3. 若当前包改了多个参数，立即走控制变量法
4. 在报告中明确写出：
   - 哪个包
   - 哪组参数
   - 什么动作触发重启
   - 判定为 `FAIL`

## 6. `FAIL / BLOCK` 的处理规则

### 不能猜，只能证明

组合包里某个功能点出现 `FAIL` 或 `BLOCK` 时：

- 不能直接断言“就是这个参数坏了”
- 必须把其他参数全部恢复默认
- 只保留该问题参数和最小依赖
- 重新打包、重新烧录、重新验证该功能点

### 控制变量包的目的

- 证明问题是参数自身导致
- 还是由其他组合参数干扰导致
- 还是测试逻辑/执行过程导致

### `BLOCK` 也必须复测

- `BLOCK` 不是终点
- 若是组合包里的阻塞项，同样要拆成单项或最小依赖包继续验证
- 只有平台外部故障、token 失效、硬件链路失联等不可继续执行的情况，才保留为最终 `BLOCK`

## 7. 异常分类

### 打包异常

- `token/auth` 异常
  - 记 `BLOCK`
  - 先修 token，再继续
- 语种与 `vcn` 不匹配
  - 记配置错误
  - 不记为平台通用故障
- 参数结构错误、算法词表结构错误
  - 记配置或脚本问题
- 编译内存超限
  - 说明逻辑通了，但当前容量不够

### 执行异常

- 配置断言失败
  - 说明参数未入包或映射错误
- 烧录失败
  - 单独记录，不进入功能通过结论
- 预处理失败
  - 停止测试，记 `FAIL` 或 `BLOCK`
- 运行态功能失败
  - 记 `FAIL`
- 协议打印异常
  - 允许按专项规则重试
  - 重试后仍异常则明确写“协议打印异常”
- 设备自行重启
  - 一律 `FAIL`
- 测试逻辑问题
  - 必须修正测试逻辑后重跑，不能把结果记到固件头上

## 8. 结果目录规则

### 一包一目录

每打一个包，都必须有一个对应结果目录；目录名就是报告中的“执行包”名称。

建议结构：

```text
scripts/result/{产品标号}/{批次目录}/result/
├── 包01-基础中值稳定包0413xxxx/
│   ├── 固件zip
│   ├── burn.log
│   ├── serial_raw.log
│   ├── test_tool.log
│   ├── testResult.xlsx
│   ├── test_report.html
│   └── 配置断言结果
├── 包02-左边界组合包0413xxxx/
└── 包03-控制变量-某功能0413xxxx/
```

规则：

- 一个包只放一个目录
- 固件、日志、结果文件必须在同一目录里
- 最终附件统一压成 `result.zip`
- `result.zip` 里只放实际执行过的目录

## 9. 报告与邮件规则

### 报告口径

报告只要求测试人员快速看懂：

- 哪些功能点 `PASS`
- 哪些功能点 `FAIL`
- 哪些功能点 `BLOCK`
- `FAIL / BLOCK` 是什么原因

### 写法要求

- 不要只写“左边界包”“右边界包”
- 必须写实际参数值
- 必须写异常现象，不要只写内部字段名
- 算法配置结果必须单独列出
- `FAIL / BLOCK` 必须写明：
  - 哪个包
  - 配置参数值
  - 执行动作
  - 观察到的异常
  - 与预期不符在哪里
  - 是否已做控制变量复测，以及复测结论

### 邮件发送前检查

1. 当前产品所有支持功能点都已有最终结论
2. 所有组合包中的 `FAIL / BLOCK` 都已做过必要的控制变量复测
3. 所有重启问题都已单独标红，不存在被吞掉的重启
4. 执行包名称与结果目录名完全一致
5. `result.zip` 中目录、固件、日志、结果文件一一对应
6. 正文不使用内部脚本术语堆砌

## 10. 执行底线

- 不得擅自替换用户要求的产品、语言、版本
- 不得因失败另起同产品的新周标
- 不得把未验证清楚的功能点写成 `PASS`
- 不得把设备自行重启写成 `PASS`
- 不得把测试逻辑问题写成固件问题
- 也不得把真实固件问题误写成“测试问题”
